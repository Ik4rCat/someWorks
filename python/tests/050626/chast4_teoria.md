# Теория — Часть 4 (050626)

## 1. Чем unit-тестирование отличается от интеграционного? Как оформляется баг-репорт?

**Unit-тестирование** — проверка отдельной функции/метода/класса в изоляции от остального кода. Зависимости заменяются моками. Быстро, легко локализовать ошибку.

**Интеграционное тестирование** — проверка взаимодействия нескольких компонентов вместе (например, функция + БД, или несколько модулей). Медленнее, сложнее отладить, но проверяет реальные связи.

| | Unit | Integration |
|--|------|-------------|
| Область | Один модуль | Несколько модулей |
| Скорость | Быстро | Медленно |
| Изоляция | Полная (моки) | Частичная или без |
| Сложность отладки | Низкая | Высокая |

**Баг-репорт** включает поля:
- **ID** — уникальный номер
- **Заголовок** — краткое описание проблемы
- **Серьёзность (Severity)** — критическая / высокая / средняя / низкая
- **Приоритет (Priority)** — как срочно надо исправить
- **Шаги воспроизведения** — пошагово как получить баг
- **Ожидаемый результат** — что должно было произойти
- **Фактический результат** — что произошло на самом деле
- **Окружение** — ОС, версия ПО, браузер
- **Вложения** — скриншоты, логи

## 2. Что такое тестирование требований и зачем оно проводится? Основные методы отладки

**Тестирование требований** — проверка самих требований до начала разработки: корректны ли они, полны, непротиворечивы, реализуемы. Проводится чтобы:
- Выявить ошибки в требованиях до того, как они превратились в код
- Убедиться, что требования тестируемы (можно написать тест-кейс)
- Снизить стоимость исправления — ошибка в требованиях на этапе разработки стоит в 10–100 раз дороже

**Методы отладки:**
- **Print-отладка** — вывод значений переменных в консоль, быстро для простых случаев
- **Пошаговая отладка (Debugger)** — выполнение кода по шагам, просмотр стека и переменных
- **Бинарный поиск** — комментирование половины кода для сужения области ошибки
- **Анализ стека вызовов (Stack trace)** — чтение traceback для понимания где упало
- **Логирование** — запись событий в файл для анализа поведения в production
- **Rubber duck debugging** — объяснение кода вслух, часто помогает самому найти ошибку
