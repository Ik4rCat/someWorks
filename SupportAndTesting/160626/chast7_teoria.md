# Часть 7 — Теория (16.06.26)

## 1. Типы тестовых данных в unit-тестировании — три типа

**1. Граничные значения (Boundary Values)**  
Данные на краях допустимого диапазона. Большинство ошибок скрыто именно там.  
Пример для `validate_age(18..100)`: значения 17, 18, 100, 101.

**2. Эквивалентные разделения (Equivalence Partitions)**  
Группы данных, которые программа обрабатывает одинаково. Достаточно одного теста от каждой группы.  
Пример: для validate_age — группа "валидный возраст" (30, 50, 80), группа "слишком молодой" (0, 10, 17), группа "неверный тип" (None, "25", 18.5).

**3. Специальные / угловые случаи (Edge Cases)**  
Данные, которые могут вызвать неожиданное поведение: пустые строки, null, 0, отрицательные числа, очень большие числа, Unicode.  
Пример: `char_count("")` → `{}`, `countEven([])` → `0`.

## 2. Критерии качества требований — три критерия

**1. Однозначность (Unambiguous)**  
Требование должно трактоваться только одним способом. Плохо: "система должна быть быстрой". Хорошо: "время отклика не превышает 200 мс при нагрузке 1000 запросов/с".

**2. Проверяемость (Verifiable)**  
Должна существовать возможность написать тест, который однозначно подтвердит или опровергнет выполнение требования. Непроверяемое требование нельзя включить в тест-план.

**3. Полнота (Complete)**  
Требование должно описывать все сценарии, включая граничные и исключительные. Если не указано что делать при некорректном вводе — требование неполное.

## 3. Тест-репорт — что это и какую информацию содержит

**Тест-репорт** — документ, фиксирующий результаты выполнения тестирования за определённый период или итерацию.

**Содержит:**
- **Сводка**: сколько тестов запущено, прошло, упало, пропущено
- **Список дефектов**: какие тесты упали и почему (с трассировкой или скриншотом)
- **Покрытие**: какие модули/функции протестированы
- **Версия продукта и окружение**: ОС, версия Python/компилятора, дата запуска

**Зачем нужен:**  
Позволяет команде и менеджменту видеть текущее состояние качества продукта. Даёт основание для решения — можно ли выпускать релиз.

---

## 4. Основные виды тестовой документации — четыре вида

| Вид | Что содержит | Зачем нужен |
|-----|-------------|------------|
| **Тест-план** | Цели, объём, стратегия, ресурсы, сроки тестирования | Определяет что, как и когда будет тестироваться |
| **Тест-кейс** | Шаги, входные данные, ожидаемый результат для конкретного сценария | Воспроизводимая проверка конкретного поведения системы |
| **Чек-лист** | Список проверок без детальных шагов | Быстрая проверка основных функций, удобен при регрессии |
| **Тест-репорт** | Результаты выполнения тестов, найденные дефекты, статистика | Фиксирует состояние качества продукта для команды и менеджмента |
