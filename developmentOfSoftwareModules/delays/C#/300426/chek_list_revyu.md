# Чек-лист код ревью — 30.04.26

## Критерии

| # | Критерий | Описание |
|---|----------|----------|
| 1 | **Название переменных и функций** | Переменные названы по стилю компании, соответствуют содержимому. Функции названы по тому, что делают. Классы отражают реальные бизнес-объекты |
| 2 | **Читаемость кода** | Код читается легко. Функция сложная и занимает 100 строк → разбить на меньшие. Если логику можно реализовать проще → переписать |
| 3 | **Соответствие требованиям** | Реализованный функционал соответствует задаче. Исполнитель не упустил важное и не добавил лишнее |
| 4 | **Комментарии в коде** | Комментарии не избыточны, поясняют предметную область или неочевидные вещи. В контрактах и бизнес-объектах есть соответствие бизнес-термину |
| 5 | **Стиль кода** | Перед ревью lint пройден. Правки по стилю сводятся только к тому, что lint не покрывает |
| 6 | **Повторное использование (DRY)** | Соблюдается принцип DRY. Функции, которые могут использоваться повторно — спроектированы соответственно. PR не изобретает велосипед |
| 7 | **Поддержка разных окружений** | Конфигурации окружений не захардкожены в коде. Код работает на разных окружениях (develop / production) и поддерживает несколько instance для горизонтального масштабирования |
| 8 | **Производительность** | Сложные вычисления вынесены в отдельные процессы, не блокируют приложение. Нет узких мест, которые дадут просадку при росте данных или нагрузки |
| 9 | **Тесты** | Тесты соответствуют договорённости о покрытии. Если функционал содержит сложную логику — есть unit-тесты |
