# GRASP Patterns

**GRASP** — General Responsibility Assignment Software Patterns.  
Автор: Craig Larman, книга *Applying UML and Patterns*.

Это не паттерны реализации как GoF, а **принципы распределения обязанностей** между классами при проектировании.

---

## 1. Information Expert (Информационный эксперт)

**Проблема:** кто должен отвечать за вычисление или хранение информации?

**Решение:** назначить обязанность тому классу, который уже обладает необходимой информацией.

**Пример:** нужно посчитать итог продажи.
- `Sale` знает список `SaleItem` → считает итог
- `SaleItem` знает количество → считает подытог
- `ProductSpec` знает цену → возвращает цену

Все три — информационные эксперты для своей части.

```
Sale ──────────── SaleItem ──────────── ProductSpec
getTotal()        getSubtotal()         getPrice()
```

## 2. Creator (Создатель)

**Проблема:** кто должен создавать экземпляры класса A?

**Решение:** назначить создание класса B, если B:
- содержит/агрегирует объекты A
- активно использует A
- имеет данные для инициализации A

**Пример:** `Sale` агрегирует `SaleItem` → `Sale` создаёт `SaleItem`.

**Плюсы:** не повышает связанность — созданный объект виден только создателю.  
**Минусы:** если создание сложное (зависит от внешних условий) — лучше использовать Abstract Factory.

---

## 3. Controller (Контроллер)

**Проблема:** кто обрабатывает входные системные события?

**Решение:** делегировать обработку событий специальному классу-контроллеру, который не относится к UI.

**Виды контроллеров:**
- **Фасадный** — представляет всю систему, подходит при малом числе событий
- **Контроллер прецедента** — отдельный контроллер для каждого use case

**Плюсы:** улучшает повторное использование; накапливает информацию о последовательности событий.  
**Минусы:** контроллер может стать перегруженным.

---

## 4. Low Coupling (Низкая связанность)

**Проблема:** как минимизировать зависимости между классами?

**Решение:** распределять обязанности так, чтобы связи между классами были минимальны.

Высокая связанность — плохо:
- изменение одного класса ломает другие
- класс сложно переиспользовать
- сложно понять класс без знания всех связей

**Применять совместно** с High Cohesion — они дополняют друг друга.

---

## 5. High Cohesion (Высокое зацепление)

**Проблема:** как не допустить, чтобы объект делал слишком много несвязанных вещей?

**Решение:** класс должен выполнять только тесно связанные между собой обязанности.

**Плюсы:** классы просты в поддержке и повторном использовании.  
**Минусы:** для распределённых серверных объектов иногда выгоднее иметь несколько крупных объектов со слабым зацеплением ради быстродействия.

---

## 6. Polymorphism (Полиморфизм)

**Проблема:** как обрабатывать альтернативное поведение на основе типа без цепочек if/switch?

**Решение:** распределить обязанности через полиморфные операции; каждая внешняя система получает свой интерфейс.

**Плюсы:** систему легко расширять и модернизировать.  
**Минусы:** не стоит злоупотреблять — добавлять интерфейсы «про запас» для неизвестных ситуаций.

---

## 7. Pure Fabrication (Искусственный)

**Проблема:** какой класс обеспечивает High Cohesion и Low Coupling, когда ни один класс предметной области не подходит?

**Решение:** создать искусственный класс, не представляющий понятие из предметной области, но решающий конкретную техническую задачу.

**Пример:** кто сохраняет `Sale` в БД?
- Нельзя добавить в `Sale` — нарушится зацепление, вырастет связанность
- Решение: создать класс `PersistenceStorage` — искусственная сущность

**Плюсы:** высокое зацепление и низкая связанность у нового класса.  
**Минусы:** легко перестараться и превратить всю систему в набор искусственных объектов.

---

## 8. Indirection (Перенаправление)

**Проблема:** как избежать прямого связывания между двумя классами?

**Решение:** ввести промежуточный объект, который берёт на себя связь между компонентами.

**Пример:** класс `PersistenceStorage` из паттерна Pure Fabrication — он стоит между `Sale` и базой данных, убирая прямую зависимость.

> Многие GoF-паттерны (Adapter, Facade, Proxy) — реализации принципа Indirection.

---

## 9. Protected Variations (Устойчивый к изменениям)

**Проблема:** как спроектировать систему так, чтобы изменение одних элементов не ломало другие?

**Решение:** выявить точки нестабильности и обернуть их стабильным интерфейсом. Изменения происходят за интерфейсом — остальная система не затрагивается.

**Примеры реализации:** интерфейсы, абстрактные классы, полиморфизм, инкапсуляция — всё это Protected Variations в действии.

## Сравнительная таблица

| Паттерн | Проблема | Ключевая идея |
|---|---|---|
| Information Expert | Кто считает/хранит данные? | Тот, у кого уже есть нужная информация |
| Creator | Кто создаёт объект A? | Тот, кто агрегирует или использует A |
| Controller | Кто обрабатывает системные события? | Специальный класс-контроллер, не UI |
| Low Coupling | Как снизить зависимости? | Минимум связей между классами |
| High Cohesion | Как не перегрузить класс? | Один класс — одна зона ответственности |
| Polymorphism | Как убрать if/switch по типу? | Виртуальные методы и интерфейсы |
| Pure Fabrication | Кто выполняет техническую задачу? | Искусственный класс вне предметной области |
| Indirection | Как убрать прямую зависимость? | Промежуточный объект-посредник |
| Protected Variations | Как защититься от изменений? | Стабильный интерфейс над нестабильным кодом |
