# Design Patterns — Analysis Table

## Порождающие (Creational)

| Паттерн | Назначение | Плюсы | Минусы |
|---|---|---|---|
| **Factory Method** | Интерфейс создания объектов, тип определяется в подклассах | Система независима от типов объектов; легко добавлять новые типы | Для каждого типа нужна своя фабрика — растёт число классов |
| **Abstract Factory** | Создание семейств взаимосвязанных объектов | Легко менять семейство целиком; объекты из разных семейств не смешиваются | Сложно добавлять новые типы продуктов — нужно менять все фабрики |
| **Builder** | Поэтапное конструирование сложного объекта | Контроль над процессом сборки; разные представления одного алгоритма | ConcreteBuilder жёстко связан с продуктом |
| **Prototype** | Создание объектов через клонирование прототипа | Клиент не знает конкретных классов; гибкое добавление типов через реестр | Глубокое копирование может быть сложным |
| **Singleton** | Единственный экземпляр класса с глобальным доступом | Контролируемый единственный экземпляр; легко адаптировать под нужное число экземпляров | Сложно тестировать; скрытые зависимости; проблемы при многопоточности |

## Структурные (Structural)

| Паттерн | Назначение | Плюсы | Минусы |
|---|---|---|---|
| **Adapter** | Адаптирует несовместимый интерфейс класса к нужному | Разделяет интерфейс и реализацию; принцип единственной ответственности | Увеличивает общую сложность кода |
| **Decorator** | Динамически добавляет объекту новое поведение | Гибкое расширение без наследования; декораторы комбинируются в любом порядке | Много мелких объектов; сложно удалить конкретный декоратор из цепочки |
| **Facade** | Упрощённый единый интерфейс к сложной подсистеме | Изолирует клиентов от компонентов подсистемы; упрощает использование | Фасад рискует разрастись в «класс-бог» |
| **Proxy** | Контролирует доступ к объекту | Ленивая инициализация; логирование; защита доступа | Усложняет код; возможна задержка первого ответа |
| **Composite** | Единообразная работа с деревьями объектов | Клиент работает с деревом и листом одинаково | Сложно ограничить компоненты дерева по типу |
| **Bridge** | Отделяет абстракцию от реализации | Абстракция и реализация меняются независимо друг от друга | Усложняет код при простых иерархиях |
| **Flyweight** | Экономия памяти при множестве похожих объектов | Резко снижает потребление памяти при массовом создании объектов | Сложнее читать код; нельзя хранить внешнее состояние внутри объекта |

## Поведенческие (Behavioral)

| Паттерн | Назначение | Плюсы | Минусы |
|---|---|---|---|
| **Strategy** | Взаимозаменяемые алгоритмы во время выполнения | Замена алгоритмов без изменения контекста; устраняет условные операторы | Клиент должен знать о существующих стратегиях |
| **Observer** | Автоматическое уведомление зависимых объектов при изменении | Слабая связанность; динамическая подписка и отписка | Непредсказуемый порядок уведомлений; возможны утечки памяти |
| **Command** | Инкапсуляция запроса как объекта | Поддержка undo/redo; очереди команд; логирование операций | Много мелких классов |
| **State** | Объект меняет поведение при смене состояния | Устраняет громоздкие if/switch; каждое состояние в отдельном классе | Много классов при большом количестве состояний |
| **Template Method** | Скелет алгоритма в базовом классе, шаги — в подклассах | Устраняет дублирование; фиксирует структуру алгоритма | Жёсткий скелет алгоритма сложно менять |
| **Iterator** | Перебор элементов коллекции без раскрытия её структуры | Единый интерфейс перебора разных коллекций | Избыточен если коллекция уже предоставляет доступ по индексу |
| **Chain of Responsibility** | Передача запроса по цепочке обработчиков | Слабая связанность; гибкое изменение цепочки | Запрос может остаться необработанным |
| **Mediator** | Централизованное взаимодействие объектов через посредника | Устраняет прямые связи между объектами | Посредник может разрастись в «класс-бог» |
| **Memento** | Сохранение и восстановление состояния объекта | Undo без нарушения инкапсуляции | Много памяти при частом сохранении снимков |
