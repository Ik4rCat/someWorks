# Behavioral Patterns — Real World Examples & Analysis

Для каждого паттерна: где используется в реальных системах, почему именно он, можно ли заменить.

---

## 3.2.1 Interpreter (Интерпретатор) — GoF

**Пример:** SQL-движок в базах данных (MySQL, PostgreSQL). Каждый SQL-запрос — это выражение, которое парсится и интерпретируется по грамматике.

**Почему использован:** запросы — это формальный язык с чёткой грамматикой. Interpreter позволяет строить дерево выражений и вычислять его рекурсивно.

**Можно ли заменить:** частично — на Visitor (обход дерева) или Strategy (разные стратегии выполнения). Но для полноценного разбора языка замена нецелесообразна — Interpreter создан именно для этого.

---

## 3.2.2 Iterator (Итератор) — GoF

**Пример:** `for-each` в любом языке. В C++ — `std::iterator` в STL-контейнерах (`vector`, `list`, `map`). В Java — `Iterator<T>`. В Python — протокол `__iter__`/`__next__`.

**Почему использован:** клиент хочет перебрать элементы коллекции, не зная её внутреннего устройства. `vector` и `list` хранят данные по-разному, но перебираются одинаково.

**Можно ли заменить:** для простых массивов — да, обычным индексом. Для сложных структур (дерево, граф) — нет, Iterator незаменим: он скрывает алгоритм обхода.

---

## 3.2.3 Command (Команда) — GoF

**Пример:** система undo/redo в текстовых редакторах (VS Code, Word). Каждое действие — объект команды с методами `execute()` и `undo()`. Также: очереди задач в системах (Git staging area — набор команд перед коммитом).

**Почему использован:** нужно откладывать, отменять, логировать и повторять операции. Command инкапсулирует запрос как объект — его можно хранить и манипулировать им.

**Можно ли заменить:** Strategy — если не нужна история. Но для undo/redo Command незаменим: только он хранит состояние для отмены.

---

## 3.2.4 Observer (Наблюдатель) — GoF

**Пример:** event-системы везде. В браузере — `addEventListener`. В React — `useState` уведомляет компоненты об изменении. В играх на Unity — система событий `UnityEvent`. В бэкенде — Kafka/RabbitMQ (pub/sub).

**Почему использован:** один объект меняется — нужно уведомить множество других без жёсткой связи между ними. Observer даёт слабую связанность.

**Можно ли заменить:** Mediator — если уведомления сложные и нужна координация. EventBus — надстройка над Observer. Полностью заменить сложно: Observer — основа всех реактивных систем.

---

## 3.2.5 Don't Talk to Strangers (Закон Деметры) — GRASP

**Пример:** в Spring (Java) контроллер не обращается напрямую к репозиторию через несколько слоёв. Только через сервис: `Controller → Service → Repository`. Каждый слой знает только о соседнем.

**Почему использован:** снижает связанность. Если цепочка `a.getB().getC().doWork()` — это нарушение, потому что A знает о B, C и их методах.

**Можно ли заменить:** Facade или Indirection решают ту же проблему структурно. Но «не разговаривай с незнакомыми» — это принцип, а не паттерн реализации, его нельзя «заменить».

---

## 3.2.6 Visitor (Посетитель) — GoF

**Пример:** компиляторы — обход AST (abstract syntax tree). Разные визиторы делают разные проходы: один считает типы, другой генерирует код, третий оптимизирует. В Java — `javax.lang.model` visitor API.

**Почему использован:** нужно добавлять новые операции над иерархией классов, не меняя сами классы. Компилятор знает грамматику — она стабильна, а операций над ней много.

**Можно ли заменить:** Iterator — если нужен только перебор. Strategy — если операция одна. Visitor незаменим когда операций много, а структура стабильна.

---

## 3.2.7 Mediator (Посредник) — GoF

**Пример:** авиадиспетчерская система — самолёты не общаются напрямую, только через диспетчера. В UI-фреймворках — `EventBus` или `ViewModel` в MVVM. Чат-сервер: пользователи не знают друг о друге напрямую.

**Почему использован:** при N объектах прямые связи дают O(N²) зависимостей. Mediator сводит их к O(N).

**Можно ли заменить:** Observer — если связи простые (один ко многим). Facade — если нужен упрощённый интерфейс к подсистеме. Для двунаправленного сложного взаимодействия Mediator лучше.

---

## 3.2.8 State (Состояние) — GoF

**Пример:** конечный автомат заказа в интернет-магазине: `новый → оплачен → отправлен → доставлен → закрыт`. Каждое состояние — свой класс с разным поведением. Также: TCP-соединение (LISTEN → SYN_SENT → ESTABLISHED → CLOSE_WAIT).

**Почему использован:** поведение объекта радикально меняется в зависимости от состояния. Без State — огромные switch/if по всему коду.

**Можно ли заменить:** Strategy — если состояния не переключают друг друга сами. Для полноценного FSM (конечного автомата) State незаменим.

---

## 3.2.9 Strategy (Стратегия) — GoF

**Пример:** алгоритмы сортировки в Java — `Collections.sort()` принимает `Comparator` (стратегию сравнения). В играх — поведение ИИ: одна стратегия атакует, другая убегает. В платёжных системах — разные провайдеры оплаты.

**Почему использован:** алгоритм нужно менять на лету без изменения клиента. Устраняет условные операторы.

**Можно ли заменить:** Template Method — если алгоритм фиксирован, меняются только шаги. State — если выбор стратегии зависит от состояния объекта. Strategy проще и гибче для замены алгоритмов.

---

## 3.2.10 Memento (Хранитель) — GoF

**Пример:** undo в графических редакторах (Photoshop — история слоёв). Git — каждый коммит это Memento состояния репозитория. Сохранения в играх (save/load).

**Почему использован:** нужно восстанавливать состояние объекта без нарушения инкапсуляции — никто снаружи не должен знать о внутреннем устройстве.

**Можно ли заменить:** Command с undo — альтернатива для простых случаев. Serialization — для сохранения на диск. Но Memento чище: снимок хранит ровно то, что нужно для восстановления.

---

## 3.2.11 Chain of Responsibility (Цепочка обязанностей) — GoF

**Пример:** middleware в веб-фреймворках (Express.js, Django). Запрос проходит цепочку: аутентификация → авторизация → валидация → обработчик. Также: обработка исключений (catch-блоки), фильтры в Java Servlet.

**Почему использован:** нужно динамически выстраивать цепочку обработчиков. Каждый обрабатывает своё или передаёт дальше. Отправитель не знает, кто именно обработает.

**Можно ли заменить:** Command + очередь — если порядок фиксированный. Decorator — если каждый обработчик обязателен. Chain of Responsibility незаменим когда цепочка динамическая и обработчик может прервать её.

---

## 3.2.12 Template Method (Шаблонный метод) — GoF

**Пример:** жизненный цикл компонентов в React (`componentDidMount`, `render`, `componentWillUnmount`) — скелет фиксирован фреймворком, реализация в подклассе. В Java — `HttpServlet.service()` вызывает `doGet()`/`doPost()`.

**Почему использован:** алгоритм фиксирован, но отдельные шаги варьируются. Фреймворк контролирует скелет, разработчик — детали.

**Можно ли заменить:** Strategy — если весь алгоритм нужно менять целиком, а не отдельные шаги. Template Method лучше когда большая часть логики общая.

---

## 3.2.13 High Cohesion (Высокое зацепление) — GRASP

**Пример:** микросервисная архитектура — каждый сервис отвечает за одну бизнес-область (OrderService, PaymentService, UserService). В UNIX — утилиты делают одно дело хорошо (`ls`, `grep`, `sort`).

**Почему использован:** класс/модуль с одной ответственностью легче понять, тестировать и переиспользовать.

**Можно ли заменить:** это принцип, а не паттерн реализации. Достигается через Single Responsibility Principle. Нельзя «заменить» — можно нарушить, получив «класс-бог».

---

## 3.2.14 Controller (Контроллер) — GRASP

**Пример:** MVC-фреймворки — `UserController`, `OrderController` в Spring/Django/Rails. Контроллер принимает HTTP-запрос и делегирует бизнес-логику сервисам.

**Почему использован:** нужна единая точка входа для обработки системных событий вне UI-слоя. Улучшает переиспользование и тестируемость.

**Можно ли заменить:** Facade — если нет состояния между запросами. Mediator — если контроллеров много и они взаимодействуют. В большинстве веб-приложений Controller незаменим.

---

## 3.2.15 Polymorphism (Полиморфизм) — GRASP

**Пример:** драйверы устройств в ОС — единый интерфейс `read()`/`write()`, разные реализации для HDD, SSD, USB. В игровых движках — `GameObject.Update()` вызывается для всех объектов, каждый реализует по-своему.

**Почему использован:** устраняет ветвления по типу. Добавление нового типа не требует изменения существующего кода (Open/Closed Principle).

**Можно ли заменить:** Strategy — если поведение меняется динамически. Visitor — если операций много. Полностью заменить нельзя: полиморфизм — фундамент ООП.

---

## 3.2.16 Pure Fabrication (Искусственный) — GRASP

**Пример:** классы-репозитории (Repository pattern) — `UserRepository`, `OrderRepository`. Они не из предметной области (нет «репозитория» в бизнесе), но нужны для работы с БД. DAO (Data Access Object) — аналогично.

**Почему использован:** ни один класс предметной области не должен знать о деталях хранения данных. Искусственный класс берёт техническую задачу на себя.

**Можно ли заменить:** Facade — если нужен упрощённый интерфейс к подсистеме. Indirection — принцип тот же. Для работы с БД Pure Fabrication (Repository) — стандарт индустрии.
